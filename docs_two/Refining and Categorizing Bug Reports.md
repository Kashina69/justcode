You have hit on exactly the core challenge of building and using AI coding agents! Managing context efficiently—often called "context rot" or "token bloat"—is the biggest hurdle when working with large codebases.

Your two ideas are actually exactly aligned with the current industry best practices. Here is a breakdown of what works best and how you can implement it:

### 1. The "Project Index" Approach (Highly Recommended)
Your idea of having a `project.md`, `architecture.md`, or a `docs/` folder is the **gold standard** for agentic coding. 

Instead of forcing the AI to blindly grep through 1,000,000 lines of code to understand how things connect, you maintain a heavily condensed, human-readable map of the project. This file should contain:
*   **Business Logic & Flow:** A high-level explanation of what the app does.
*   **Architecture & File Structure:** "Database schemas are in `/supabase`, UI components in `/agency/components`, routing in `/agency/app`."
*   **Design Patterns:** "We use Tailwind for styling, Context API for state, and all notifications must be fire-and-forget background tasks."

**How the AI uses it:** The AI reads this 500-line file first. Using that "map," it knows *exactly* which specific 3 files it needs to open to fix a bug, skipping the need to grep blindly through the codebase.

### 2. Context Caching (Your "Base Session" Idea)
Your idea of replicating a "base chat session" is brilliant, and it is exactly how modern AI APIs are solving this problem! It is called **Prompt Caching** (supported by Gemini and Claude).

In an agentic harness, you don't actually need to copy the session history. Instead, you load your `project.md`, database schema, and core utility files into a "System Prompt." The AI API caches this massive block of tokens in its memory for a fraction of the cost. Every new chat session you start against that cache will instantly "remember" your entire project without having to re-read it, saving up to 90% of token costs per turn.

As an agent (Antigravity), I actually use a system very similar to your "skill files" idea behind the scenes! I maintain a local `knowledge/` folder with summaries of past conversations and resolved bugs. Before I start a new task, I read these distilled "Knowledge Items."
